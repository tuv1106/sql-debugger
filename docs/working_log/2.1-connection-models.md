# Task 2.1: Connection Models (Pydantic)

## Goal
Define Pydantic models for database connection parameters (Postgres, MySQL, BigQuery) with validation, plus shared response models in `common.py`.

## Files to Create
1. `local_backend/models/connection.py` — ConnectionParams base + 3 subclasses
2. `local_backend/models/common.py` — Shared response models (ConnectionSummary)
3. `tests/local_backend/test_models_connection.py` — 7 tests

## Models to Implement

### `local_backend/models/common.py`
- `DbType(str, Enum)` — `postgres`, `mysql`, `bigquery`
- `ConnectionSummary(BaseModel)` — `id`, `name`, `db_type` (password-free summary for list endpoints)

### `local_backend/models/connection.py`
- `ConnectionParams(BaseModel)` — base class
  - Fields: `name: str`, `db_type: DbType`
  - Abstract-like: subclasses define their own fields
- `PostgresConnectionParams(ConnectionParams)` — `host`, `port` (1-65535), `database`, `username`, `password`
- `MySQLConnectionParams(ConnectionParams)` — `host`, `port` (1-65535), `database`, `username`, `password`
- `BigQueryConnectionParams(ConnectionParams)` — `project_id`, `service_account_json` (str, holds the JSON content)
- `db_type` is auto-set per subclass (e.g., Postgres always has `db_type=DbType.POSTGRES`)

### Validation Rules
- `name`: required, non-empty string
- `host`: required, non-empty string (Postgres/MySQL)
- `port`: required, integer 1-65535 (Postgres/MySQL)
- `database`: required, non-empty string (Postgres/MySQL)
- `username`: required, non-empty string (Postgres/MySQL)
- `password`: required, non-empty string (Postgres/MySQL)
- `project_id`: required, non-empty string (BigQuery)
- `service_account_json`: required, non-empty string (BigQuery)
- Validation runs on model instantiation (Pydantic default) — the spec says "on create/update/test only, NOT when loading from config", but that's handled at the service layer, not the model layer. Models always validate; the config store can use `model_construct()` to skip validation when loading.

### Design Decisions
- Use Pydantic v2 `BaseModel` with field validators
- Port validation via `Field(ge=1, le=65535)`
- `db_type` set via `model_validator` or literal default on each subclass
- `ConnectionSummary` lives in `common.py` since it's used by multiple layers (ConfigStore, API)
- No Snowflake (post-MVP)

## Tests (7)

1. **Valid Postgres params** — all fields provided, model creates successfully
2. **Valid MySQL params** — all fields provided, model creates successfully
3. **Valid BigQuery params** — project_id + service_account_json, creates successfully
4. **Missing required fields** — omitting `host` on Postgres raises ValidationError
5. **Port below range** — port=0 raises ValidationError
6. **Port above range** — port=70000 raises ValidationError
7. **Invalid db_type** — passing unknown db_type string raises ValidationError

## Retrospect

Implementation followed the plan with two deviations:
- **Dropped `needs_reauth`** from `ConnectionSummary` — per developer feedback, this belongs to the ConfigStore layer (Task 2.4), not the model definitions.
- **Added 3 type validation tests** (one per DB type) per developer request, bringing total from 7 to 10 tests.
- `db_type` was set as a simple `Field(default=...)` on each subclass rather than `model_validator` — simpler and sufficient.
